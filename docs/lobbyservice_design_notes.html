Lobby Service Design Notes

Chris Greenhalgh, 2010-08-17

Client interaction

There are two main client interactions:

  1. list available games (for this client)
  2. join specific game

The first interaction is based on a previously downloaded (probably statically served) lobby service description file. 

See lobbyservice_interaction_notes.html for more details of the proposed interaction.

Data Model

Main entities:

  • Client - device, typically
  • User - person with some sort of account [e.g. for admin / orchestration, or instead of ticketing]
  • Game Server - specific game server, e.g. droolstest instance [for internal management, e.g status tracking, instance creation]
  • Game Template - information about a family of similar game instances, which might be useful to create them, c.f. a droolstest template. May be linked to a particular Game Server or perhaps a type of Game Server. [for internal management and instance creation/lifecycle; also user concept c.f. film and showing]
  • Game Instance - a specific (potentially multi-user) game instance, e.g. a droolstest "session"
  • Game Instance Slot - a particular client's involvement with a particular game instance, c.f. droolstest "conversation"
  • Ticket - a lobby-generated credential which allows a Client access to a Lobby and/or Game Instance
  • Client Filter - client black/white list
  • Network Filter - network black/white list
And some key Protocol objects:
  • Client Description - e.g. IMEI, user, capabilities/application versions, etc.
  • Lobby Response - e.g. challenge, game(s), ...
  • Partial Game Description (or Game Pattern) - (partial) description of a game (or set of related games)
  • Game Server Referral - information given to the client in on successful join request to allow it to enter the actual game.

Use Cases

Initial use case(s):

Use case: Exploding Places July test

Loan devices. Custom (game-specific) application. 4 (?) pre-scheduled game instances. 10 (?) slots per game instance. Single game server. Single game instance active at one time (may be same instance reused).

=>

Single GameServer. Small number of per-configured Game Instances (possibly from a single Game Template). Each with a fixed (?) set of Slots. Possibly with associated Tickets.

Use case: Droolstest / Hyperplace development

Own(ish) devices. Generic application download (could be from marketplace). 

=>

Single Game Server and Game Template per development / developer. Game Instance created on demand?  Slots created/allocated on demand.
